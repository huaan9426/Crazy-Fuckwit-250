# 250 万清空挑战开发计划

## 目标

先做网页版本，目标是易发布、易迭代、可分享。后端负责内容、成绩和排行榜，前端负责高频交互和渲染。后续如果转小程序，API 和数据结构要尽量能复用，但第一版不为小程序提前堆复杂适配层。

## 产品方向

默认模式是「混沌人生」。

玩家拿到 2,500,000 元，目标是尽快花光。游戏不追求公平竞速，追求快节奏、荒诞消费、随机意外和结算战报。

核心体验：

- 每轮一次性展示 9 张商品卡，玩家从众多消费里选择其一，然后刷新下一轮
- 商品卡有默认展示时间，不点击会错过这一轮货架
- 普通场景以小额和中额消费为主，高消费卡低概率出现
- 高消费场景按阶段限时出现，进入后才明显提高大额消费密度
- 倍率卡随货架出现和过期，倍率越高，允许使用的单价和总额越低
- 退款、中奖、赔付会反向加钱
- 利息延迟启动。当前规则是 60 秒后开始计息，第 70 秒收到第一笔；余额越少，利息百分比越高
- 生病、情绪、误操作、意外事件会改变节奏
- 钱花光后生成可传播战报

## 开发约束

- 首版做网页 + 后端 API + PostgreSQL。
- 不做登录。
- 做公开排行榜。
- 用户开局输入用户名，用户名全局唯一。
- 排行榜只展示名次、用户名、用时、单笔最高消费。
- 商品、场景、事件、状态由后端提供。
- 不接真实支付。
- 不做真实赌博暗示。
- 不接地图、商品比价、真实电商接口。
- 商品价格全部标注为娱乐估算。
- 死亡、疾病、抑郁相关内容默认进入黑色幽默处理，不写成医学建议，不嘲笑真实患者。
- 代码先服务网页发布，不做抽象过度的小程序兼容层。

## 技术选择

- Vite
- TypeScript
- 原生 DOM
- CSS
- Go HTTP API
- PostgreSQL（后续持久化阶段接入）
- 本地 TS 内容数据仅作为开发样例和兜底

暂不使用 React/Vue。理由是首版交互可以由简单状态机和 DOM 渲染完成，先减少框架成本。限制是后续页面复杂后，可能需要重构到组件框架。

PostgreSQL 作为首版主库。理由是内容、事件、成绩和排行榜都需要后端持久化，后续公开发布和并发写入比 SQLite 更稳。限制是本地开发和部署复杂度高于 SQLite。

首版暂不加 Redis。若访问量明显上升，再给排行榜、内容包和限流加缓存。

后端当前改为 Go 实现。首个可运行版本优先使用 Go 标准库打通 API，避免在业务很小的时候引入过重框架；接入 PostgreSQL 时再补迁移和持久化层。

## 首版功能闭环

1. 显示余额、计时器、已花金额、意外收入。
2. 每轮一次性展示 9 张商品卡，玩家选择其中 1 张结算，然后刷新下一轮。
3. 普通场景的大额消费卡出现概率控制在低位，整体低于 10%；进入拍卖、面具舞会等特定场景后才提高高消费密度。
4. 商品卡有默认刷新时间。玩家不选择时，当前货架会过期并刷新，稀有大额消费会被错过。
5. 倍率卡随货架一起出现和过期。倍率越高，允许使用的商品单价和本次消费总额越低，避免一个倍率按钮直接结束游戏。
6. 利息延迟启动。当前规则是 60 秒后开始计息，第 70 秒收到第一笔；余额越少，利息百分比越高，利息计入意外收入，用来阻碍玩家过快清空余额。
7. VISA 技能自动选择当前 9 张卡中消费金额最高且当前余额买得起的一张，当前平衡算法默认约 23 秒后扣款，技能冷却默认跟随阶段长度。
8. 清空购物车随机选择当前 9 张卡中的 3 张消费卡，当前平衡算法默认等待 7 秒后统一结算；结算等待期间不刷新新卡，技能冷却默认跟随阶段长度。
9. 一局主线目标时长约 7 分钟，按 12 个约 35 秒的阶段推进，阶段变化时切换消费场景压力；11 分钟是硬结算时间，届时如果仍有余额，则进入“少花钱多挣钱”的隐藏结局反馈。
10. 购买后触发随机事件。
11. 状态效果影响商品概率和事件概率。
12. 余额低于 1000 元进入找零阶段。
13. 余额归零或终局事件触发后生成战报。
14. 提交成绩到数据库。
15. 展示排行榜。
16. 本地保存最近战报。

## 排行榜规则

排行榜公开展示 4 个字段：

- 名次
- 用户名
- 用时
- 单笔最高消费

排序规则：

1. 余额归零成绩优先。
2. 用时越短排名越高。
3. 用时相同，单笔最高消费越高排名越高。

用户名规则：

- 用户名全局唯一。
- 不做密码。
- 不做登录态。
- 已存在用户名不能再次注册新成绩，除非后续显式加入“覆盖成绩”规则。
- 开局前的用户名占用是短期预约，不等于排行榜成绩。玩家刷新页面时可以用同一浏览器保存的预约 token 续租；如果预约后没有提交成绩，预约过期后会释放。若用户名仍有未过期预约，提交成绩时也必须带匹配 token，避免其他请求抢占这个预约名。
- 用户名需要长度限制和敏感字符过滤。

数据库内部字段不等于排行榜展示字段。首版至少保留：

- `id`
- `username`
- `duration_ms`
- `max_single_spend`
- `final_balance`
- `total_spent`
- `total_income`
- `ended_by`
- `chaos_seed`
- `created_at`

这些字段用于排序、防重复、排错和生成战报；页面仍然只展示 4 个字段。

## 后端内容规则

后端负责提供这些内容：

- 商品
- 场景
- 事件
- 状态
- 终局事件
- 游戏配置

前端可以缓存当局内容包，但不能把完整内容体系写死在前端。这样后续可以调整商品价格、事件概率、场景权重，而不需要重新发前端版本。

后端内容包还应该返回 `balanceTuning`。它不是单条商品内容，而是一局游戏的平衡参数，包括阶段长度、货架刷新时间、收银结算锁、利息启动延迟、分段利率、技能延迟、技能冷却、大额卡概率、事件概率加成和倍率规则。前端读取这些参数执行玩法，避免把关键节奏长期硬编码在浏览器端。

首版 API：

- `GET /api/content/bootstrap`：获取初始配置、商品、场景、事件、状态。
- `POST /api/users/reserve`：占用唯一用户名。
- `POST /api/runs`：提交一局成绩。
- `GET /api/leaderboard`：查询排行榜。

主要限制：

- 首版不做后台管理页面，内容先通过种子脚本写入数据库。
- 首版不做实时反作弊，只做基础字段校验、时长下限校验和频率限制预留。
- 前端仍然能伪造请求，真正上线需要服务端校验更完整的 run 摘要。

## 混沌系统设计

默认模式就是「混沌人生」，所有混沌包默认开启。游戏不追求公平，只追求高密度、快节奏、真实荒诞和战报传播。

混沌不是单纯随机扣钱，而是让玩家不断遇到现实里可能发生、普通人不常接触、或者消费场景里被低估的成本。

首版混沌包：

- 日常小额包：早餐、外卖、奶茶、便利店、会员续费、打车、停车补费。
- 大件消费包：手机、电脑、家电、家具、车、车位、装修、婚礼。
- 富人体验包：游艇日租、私人晚宴、面具舞会、拍卖会、古堡酒店、南极邮轮、欧洲自驾。
- 高端隐形成本包：押金、服务费、清洁费、保管费、仓储费、加急费、超时费、小费、保证金。
- 误操作包：拍卖误举牌、订错不可退酒店、重复下单、点错套餐、误买年卡、手滑包场。
- 返现进账包：退款、退税、运费险、押金退回、朋友还钱、平台补贴、会员返现。
- 中奖捡漏包：彩票、抽奖、盲盒隐藏款、古玩捡漏、二手转卖、年会中奖。
- 赔付包：航班延误、酒店超售、外卖异物、快递损坏、保险理赔、平台补偿。
- 健康包：牙疼、过敏、肠胃炎、急诊检查、住院押金、心脏报警、救护车费用。
- 情绪包：生气、低落、上头、攀比、焦虑囤货、购物麻木、报复消费、贤者时间。
- 社交压力包：份子钱、临时红包、请客、伴手礼、慈善捐款、家庭催买、面子升级。
- 宠物包：急诊、检查、住院、手术、进口药、营养品、保险赔付。
- 法律行政包：罚单、签证加急、证件补办、违约金、手续费、滞纳金。
- 平台规则包：自动续费、优惠券过期、退票失败、改签费、保价成功、限时补贴。
- 灾难维修包：手机摔坏、电脑进水、相机镜头摔裂、车胎爆了、空调漏水、装修返工。
- 终局包：猝死、意外身亡、心脏病急发、事故重伤、昏迷住院、破产式清零。

事件层级：

- 扣钱事件：直接减少余额。
- 返钱事件：增加余额，阻碍玩家清空。
- 状态事件：改变商品刷新和事件概率。
- 场景事件：只在特定场景出现。
- 连锁事件：购买某类商品后触发后续成本。
- 终局事件：提前结束本局并生成特殊战报。

状态效果：

- 生气：高价商品概率上升，冲动消费概率上升。
- 低落：商品刷新变慢，场景减少。
- 上头：批量购买档位提高，误操作概率上升。
- 攀比：奢侈品、礼物、请客、会籍概率上升。
- 焦虑囤货：日用品、食品、药品批量出现。
- 购物麻木：商品正常扣钱，但结算标记机械消费。
- 报复消费：高价商品刷新暴涨，返钱事件概率下降。
- 心悸：医疗事件概率上升，操作短暂停顿。
- 疲劳：刷新速度下降，快捷操作延迟。
- 好运：返钱、捡漏、赔付概率上升。
- 倒霉：损坏、罚单、加价概率上升。
- 回光返照：终局前短时间高消费商品暴涨。

真实荒诞感来源：

- 同一个消费不是只扣一次钱，而是可能带出押金、服务费、损坏费、保险费、清洁费。
- 高端场景不直接卖顶级资产，更多卖入场券、体验、定金、租赁、保证金。
- 返钱不是奖励，而是阻碍通关的反向事件。
- 健康和死亡事件默认开启，但用克制黑色幽默处理，不给医学建议，不羞辱真实病痛。
- 社交压力、平台规则、行政成本和维修翻车用于补足普通消费之外的真实摩擦。
- 结算页要记录最烦人返钱、最荒诞扣款、最大单笔消费、终局原因。

内容写法约束：

- 文案短，单条事件 1-2 秒能读完。
- 不使用真实明星、真实品牌商标或真实个人。
- 不写医学建议。
- 不把穷、病、死亡本身当笑点，笑点来自系统目标和现实成本冲突。
- 价格是娱乐估算，不伪装成市场报价。
- 富人场景要用租赁、押金、入场费、定金、服务费表达，避免写出明显不可信的购买行为。

## 首版内容规模

- 商品卡：300 个
- 场景：20 个
- 事件：80 个
- 状态：12 个
- 终局事件：6 个

首版可以先实现少量样例数据跑通，再逐步补内容。不要先手写 1000 条内容，避免规则未稳定时返工。

## 主要模块

- `src/game/types.ts`：基础类型。
- `src/game/constants.ts`：初始金额、默认模式、余额阶段。
- `src/game/api.ts`：前端 API 请求和离线兜底入口。
- `src/game/fallbackContent.ts`：Go API 未启动时的本地内容兜底。
- `src/game/rules.ts`：刷新、购买、事件触发、找零规则。
- `src/game/state.ts`：游戏状态。
- `src/main.ts`：页面入口和渲染。
- `server/`：Go API。
- `server/db/`：后续 PostgreSQL 连接、建表 SQL、种子数据。
- `shared/`：前后端共享类型和校验。
