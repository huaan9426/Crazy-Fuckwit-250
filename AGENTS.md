# AGENTS.md

## 项目边界

本仓库的主工程是仓库根目录，不是单独的 `src/`，也不是单独的 `server/`。

`src/` 是前端主体验，负责游戏画面、交互、音效入口、计时器、压力条、排行榜展示和浏览器端兜底。当前项目能不能吸引用户，主要取决于这里的体验质量。

`server/` 是 Go 后端服务，负责内容包、用户名占用、成绩提交和排行榜 API。后端不应该承担复杂游戏渲染逻辑，也不应该把前端表现细节塞进接口。

`docs/` 存放少量真正会被维护的项目文档。不要为了显得完整而新增空泛文档；新增文档必须能指导开发、验收或排错。

## 本规范来源

这份规范参考了几个高星或高相关度项目，但不会照抄它们的规则。

- `agentsmd/agents.md`：约 22.8k stars，核心思路是用一个固定、可预测的 `AGENTS.md` 文件给编码代理提供项目上下文和工作约束。
- `PatrickJS/awesome-cursorrules`：约 40.2k stars，核心思路是把 AI 编码规则按项目、框架、文件类型和工作流组织，减少反复纠偏。
- `ciembor/agent-rules-books`：约 2.1k stars，核心思路是把长期工程经验整理成可执行的 AI agent rules 和 skills。
- `yzhao062/agent-style`：约 540 stars，核心思路是约束 AI 输出的写作方式，让说明更像工程沟通，而不是堆口号。
- Go 官方 `Effective Go`、`Go Code Review Comments` 和 `Go Doc Comments`：用于约束 Go 后端代码风格、格式化、错误处理和文档注释。

这些来源只提供方向。本项目最终以当前代码、当前目标和本文件为准。

## 工作方式

每次只完成一个任务。任务没有完成前，不要顺手改 unrelated 文件，不要开启新的大重构，不要把“以后可能有用”的东西塞进代码或文档。

所有修改必须先读现有代码，再判断最小改动点。项目现在处在早期可玩原型阶段，最重要的是保证能跑、前后端概念一致、后续改动不会互相打架。

向后兼容是默认要求。已有 API 字段、页面入口、脚本命令和文档结论，除非明确要改，否则不要破坏。必须破坏时，要在进度文档里记录原因、影响范围和迁移办法。

不要用“架构更优雅”作为单独理由引入复杂抽象。只有当重复已经造成真实维护成本，或者接口边界已经稳定，才增加抽象层。

## 硬编码规则

硬编码不是完全禁止，但必须被控制。

允许短期硬编码的情况只有两类：第一类是项目启动阶段的样例数据，例如当前 Go 后端内存排行榜和内容包；第二类是不会变化的界面常量，例如画布基础尺寸。即使允许，也要能看出它只是当前阶段的临时实现。

禁止继续扩散的硬编码包括：接口地址散落在多个文件、价格和商品规则同时写在前端和后端、魔法数字直接写进复杂逻辑、视觉参数没有名字、同一份内容在多个地方手工复制却没有同步说明。

后续清理顺序是：先把魔法数字提升为命名常量，再把游戏内容统一交给后端内容包，最后把需要运营调整的数据迁移到数据库或种子数据。不要一步到位做庞大的配置系统。

## Go 后端规则

Go 后端是本项目的后端方向。除非用户明确改技术栈，不要再引入 Node 后端、Fastify 后端或多语言后端并行方案。

后端当前优先使用 Go 标准库实现 HTTP API。原因很简单：现在业务不复杂，标准库已经足够表达路由、JSON、CORS 和基础校验。等 PostgreSQL、迁移、日志、鉴权或中间件需求真实出现时，再评估是否引入额外库。

Go 代码提交前必须执行 `gofmt`。如果本机没有 Go 工具链，必须明确说明“未能格式化/未能运行 Go 测试”，不能假装已经验证。

Go API 的 JSON 字段要和 `src/game/types.ts` 保持一致。新增字段时先确认前端是否需要，删除字段时先确认是否会破坏旧页面。

Go 错误处理要保持显式。不要吞掉错误，不要把所有错误都返回成同一句话。对外响应可以简短，但服务端内部应保留足够上下文，方便排查。

Go 后端不写真实支付逻辑。本项目只模拟刷卡和购物压力，不接 Visa、银行卡、电商、地图、真实商品比价或真实支付接口。

## 注释规则

教学代码必须使用长文本、普通字词、连续语句来写注释，优先照顾初学者理解。

注释不要写成短词、口号、提纲或只给结论。遇到 `Enum`、`Pydantic`、`Protocol`、`JSON`、`token`、`context`、`goroutine`、`interface`、`type` 这类概念时，要先解释它是什么，再解释为什么这里要用它，最后说明它和前后代码怎么连接。

注释允许详细，但必须服务于理解当前代码。不要堆砌无关理论，不要为了显得专业而使用生硬术语。读者应该能通过注释知道这段代码为什么存在、输入从哪里来、输出会给谁用、失败时会发生什么。

简单代码不需要机械注释。像 `const API_PORT = 3001` 这种一眼能看懂的内容，不要写“定义 API 端口”这种废话。真正需要注释的是状态机、时间/动画计算、前后端 JSON 协议、并发或锁、兜底策略、排序规则、反作弊假设和任何会影响用户体验的非显然逻辑。

修改旧代码时，如果旧代码明显缺少解释，不要只在新代码加注释。要在同一个修改范围内补上关键上下文，让后来的人能顺着读完整段逻辑。

## 前端规则

前端当前是 Vite + TypeScript + Phaser + 原生 DOM + CSS。不要在没有明确收益前引入 React、Vue、全局状态库或 UI 组件库。

游戏第一屏必须是可玩的体验，不做营销式落地页。当前方向是高压购物、刷卡付款、清空购物车、倒计时、压力条和流水反馈，不做水果切割玩法。

前端可以有本地兜底内容，但兜底只为后端未启动时可玩。后端 API 一旦在线，内容、用户名和排行榜应优先来自后端。

涉及视觉和交互的修改，必须用浏览器验证。至少检查桌面和移动端是否能开局、计时器是否动、压力条是否动、核心按钮是否生效、页面是否出现横向溢出。

## 文档规则

文档只保留三类：项目计划、开发规范、进度记录。新增其他文档前必须先确认现有文档放不下。

进度文档必须记录事实，不写宣传语。已经完成、已经验证、已知问题、下一步建议要分清楚。不能把“代码已经写了”说成“功能已经上线”，也不能把“前端兜底可玩”说成“前后端已经在本机完整联调”。

文档和代码冲突时，优先相信代码，然后修正文档。文档不能成为代码之外的第二套幻想系统。

## 验收规则

前端修改至少运行 `npm run build`。如果改了依赖，还要运行 `npm audit --audit-level=high`。

后端修改至少运行 `gofmt`、`go test ./...`。如果 Go 工具链缺失，必须在最终说明和进度文档里写清楚。

前后端协议修改必须同时检查 `server/types.go` 和 `src/game/types.ts`。只改一边就是不完整修改。

任何时候都不要声称未验证的事情已经通过。能跑、能构建、能联调、能上线是四个不同等级，必须分开说。
